Safe ingestion is a protocol, not a folder.

This change turns kb add into a durable handoff for machine producers such as Screenpipe. KB records bytes, identity, provenance, and state. An agent still makes the semantic decision about what belongs in durable Memory.

The implementation covers the required slices from docs/SPEC-MACHINE-SAFE-INGESTION.md: machine-safe Add handoff v2, the persistent Basic Memory 0.22.1 Engine runner, public read-only schema inference/validation/diff, and dirty-index recovery after a durable B1 completion.

Why this change exists

A Screenpipe pipe can produce thousands of Markdown artifacts. Sending those files into one directory solves storage, but it does not answer the knowledge questions: Is this durable? Is it new? Does it update an existing Memory? Should one artifact become several Memories? What type is it?

The design therefore separates two kinds of work:

Mechanical facts

Readable input, exact bytes, SHA-256, producer identity, capture time, immutable raw storage, durable state, replay detection, and lineage checks are deterministic. KB owns them.

Semantic judgment

Create, update, split, merge, decline, title, summarize, tag, relate, and choose a meaningful type require context. The agent owns them.

This is the central boundary: physical ingestion may be automatic; semantic filing must remain explicit.

The intuition: a receiving dock with a claim check

Think of kb add as a receiving dock. The dock accepts one sealed package, fingerprints it, stores it without alteration, and gives the producer a claim check. It does not open the package and reorganize the library by itself.

1. ProducerSupplies one file or URL and, when available, an immutable event ID.
→
2. StageKB hashes bytes, stores raw content, and writes a pending receipt atomically.
→
3. AgentReads raw content plus existing Memories and chooses the semantic outcome.
→
4. VerifyKB checks source lineage, Memory paths, catalog entries, and completion consistency.
→
5. CompleteThe durable receipt records derived or raw-only.

If the producer retries, the claim check makes the operation idempotent. If it reuses the same producer ID for different bytes, KB fails closed instead of guessing whether the input is a revision.

Architecture and state model

The public lifecycle

# Stage exactly one producer event
kb add report.md --source screenpipe --source-id artifact-123 \
  --captured-at 2026-07-10T13:00:00Z --in work --json

# Recover the same playbook later
kb add --resume add-a1b2c3d4e5f60718293a4b5c --in work --json

# Complete with one or more derivatives
kb add --complete add-a1b2c3d4e5f60718293a4b5c \
  --memory memories/daily-summary.md --in work --json

# Or explicitly retain only the raw source
kb add --complete add-a1b2c3d4e5f60718293a4b5c \
  --no-memory --reason "No durable knowledge" --in work --json

Identity is not content equality

SituationRequired resultWhy
Same producer identity, same bytesReturn the existing handoff with replayed: true.A retry must not duplicate raw data, logs, or work.
Same producer identity, different bytesExit 65; mutate nothing.A producer ID is an immutable event contract, not a mutable filename.
Same bytes, different producer identitiesCreate distinct ingress events.Two capture events may have distinct provenance despite equal content.

Different completion for a completed handoffFail and preserve the first outcome.Recorded history must not silently change.

Completion is verification, not authorship

Before moving a handoff to completed state, KB verifies every declared Memory is under memories/, exists, is not an untouched TODO draft, cites the current raw ref in source_refs, and appears exactly once in index.md. KB does not inject the frontmatter or rewrite the Memory on the agent’s behalf.

Why the JSON envelope matters

A pipe needs one machine-readable result on a predictable channel. In JSON mode, success is exactly one compact object on stdout; controlled failure is exactly one compact object on stderr with stdout empty. The envelope includes the handoff ID, raw ref and hash, provenance, replay state, playbook, and exact next-command templates.

Why schema work comes later

Schema inference over unreviewed raw artifacts would infer the habits of the capture stream, not necessarily a useful knowledge model. Ticket 2 adds read-only infer, validate, and diff wrappers only after Ticket 1 creates a consistently typed, agent-reviewed corpus.

Behavioral contracts worth remembering

One input at a time

The command does not walk folders, expand globs, watch a filesystem, or own scheduling. A producer loops and checkpoints at its own boundary.

Raw bytes never change

The external source is untouched. The content-addressed raw copy is never overwritten, and resume/completion recompute its hash.

Errors are contracts

64 means invalid use or refs. 65 means bad/colliding/tampered state. 69 means optional engine failure after durable semantic work.

Legacy survives migration

Existing human-oriented Add flows, legacy pending records, a unique legacy raw-ref resume, and positional completion remain supported for one compatibility release.

Lineage is explicit

Every derivative carries its raw ref in source_refs. Updating a Memory appends lineage; it never erases prior source refs.

Schema remains advisory

Inference may propose a schema later. It never silently installs one or moves validation from warning to strict.

Verification ledger

Every required path has executable evidence. Tests exercise the public CLI subprocess seam; Engine-only tests use the explicitly forced subprocess seam for runner and process-tree behavior.

Deviations

Engine fallback is pinned to Basic Memory 0.22.1

The specification called for repairing the runner so a successful uvx availability check uses the same persistent command prefix, but it did not require a version pin. The conservative implementation uses uvx --from basic-memory==0.22.1 bm instead of an unversioned fallback. A directly installed bm executable is still preferred when present.

Why: this prevents an upstream release from changing the schema/search contract underneath KB and matches repository contract tests that already pin version 0.22.1. The tradeoff is deliberate upgrade work later rather than surprise drift now.

Schema uses Basic Memory's top-level capability route

The older repository guidance generalized all Engine operations as bm tool. Basic Memory 0.22.1 actually exposes schema inference, validation, and diff as top-level bm schema ... --json --local commands. ADR-0001 and the build guide now record this explicit capability exception; search remains on bm tool search-notes.

Why: using the real upstream interface is safer than inventing a nonexistent tool command. The process, pinning, normalization, and no-fork boundaries are unchanged.

Project-name collisions fail closed

When an Engine project already uses the requested KB name at another local path, KB rejects enablement. A project already pointing at the same canonical path is reused, including a lexical alias that resolves to that path.

Why: this is the most conservative integration of the later reliability contract: it prevents silent attachment to the wrong project and avoids inventing a private project name the user did not request.

Automatic schema drafting remains deferred

Ticket 3 described kb schema infer <type> --draft as possible follow-on work, not a required first contract. The integrated CLI rejects --draft with an explicit deferred message. It does implement the merge-critical part repeated across Tickets 1 and 3: a completed B1 handoff survives refresh failure, leaves .kb/engine-dirty, and the next Engine-backed command repairs it.

Why: writing a schema automatically would cross the agent-judgment boundary. The conservative choice keeps inference and drift read-only while preserving durable Markdown when the rebuildable index is unavailable.

Type checking uses an isolated modern compiler

The repository has no typecheck script, and its ambient TypeScript 4.7.4 cannot understand the installed Bun declarations. Verification therefore pins TypeScript 5.9.3 through bunx --bun --package typescript@5.9.3 tsc and checks the production source files with Bun-compatible module resolution.

Why: treating the known-stale ambient compiler as authoritative would create a false failure. Pinning the verifier is reproducible and does not add a runtime dependency or modify the package manifest.

Report-only conservative choice: remove stale onboarding claims

The previous implementation-notes.html described a different onboarding repair and claimed completed test/browser receipts unrelated to this spec. Its useful self-contained layout and accessible quiz pattern were preserved, but its claims were replaced so this report cannot be mistaken for evidence about machine-safe ingestion.

What this change does not promise

  • No directory ingestion, watcher, daemon, scheduler, or bulk migration of the existing thousands.
  • No automatic semantic routing, schema adoption, Memory authoring, deduplication, contradiction detection, or truth guarantee.
  • No rewriting raw artifacts to add metadata and no second schema registry owned by KB.
  • No public engine project flags or cloud routing.
  • No claim that schema inference can replace representative sampling and agent review.

A future legacy migration should begin with roughly 20–50 representative artifacts. That sample establishes useful types and realistic review throughput before anyone specifies a checkpointed bulk importer.
