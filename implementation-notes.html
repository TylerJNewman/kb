The intuition: onboarding is a state machine

The previous pages described a sequence of commands as if they were prose. But the product has durable states: no runtime, no KB, a staged source, a pending agent handoff, a completed handoff, and an optional search upgrade. If the page skips one state, the user sees contradictions such as “Health: ok” while pending metadata still exists.

What changed

Install truth

Every beginner surface now installs Bun, exports its path, installs kb, verifies kb --version, and checks Git before creation. The short hero preview follows the same order.

Deterministic targeting

Setup, status, search, completion, and resume examples use --in research. A probe from inside another KB proved the source still landed in research.

Explicit agent seam

The runnable setup stops after kb add. A separate handoff message tells the AI to follow the complete playbook, run the generated completion command, and return the receipt.

Recovery

Existing-KB refusal is explained as safe. Pending work points to --resume. Git-failed scaffolds are initialized, re-run through kb new for registration, then checked with status.

Accessible navigation

One hash-focus helper handles direct links, clicks, and hashchange. All fragment section headings accept programmatic focus. The visual walkthrough has ordinary same-site anchors.

Narrow contracts

A new test file checks prerequisites, terminology boundaries, setup/handoff/verify ordering, hero install order, focusability, receipt ordering, and recovery tokens—without snapshotting visual layout.

Deviations and conservative choices

Git recovery needed two registry states

The first isolated reproduction had no registry, so scan reconstruction made the retry return the safe “KB already exists” refusal. Spec review identified the missing state: with another KB already registered, the partial scaffold is invisible until kb new research is rerun, which prints Recovered KB. Final guidance covers both results, then continues to status.

Reflect test clock

The full suite exposed an existing date-dependent fixture whose hard-coded “now” had become earlier than the file created by the test. The fixture mtime is now pinned one hour before its test clock. No Reflect production logic changed.

Git transaction remains deferred

The CLI still writes scaffold files before Git initialization and can temporarily consider that folder complete. Fixing transaction/recovery semantics is a separate business-logic change; this patch documents a verified recovery path without weakening validation or widening scope.

What was preserved

  • kb start remains deterministic, read-only, text-only, and non-interactive.
  • The Basic Memory file-format compatibility contract and engine-specific names remain intact internally and at engine boundaries.
  • kb new still refuses duplicates; no silent idempotency or automatic recovery was added.
  • kb add --complete remains an explicit validated receipt; nothing auto-completes pending work.
  • No dependency, router, DOM library, visual snapshot, animation test, theme persistence, or timing test was added.
  • Pre-existing changes under .agents/, skills-lock.json, .scratch/, agent/, and tickets.md were not modified or staged by this work.
